# Terraform State Commands

> In this guide, you will learn how to list and manipulate Terraform state files using various terrafomr state commands. 
>* The state is stored in JSON format, and it is crucial not to edit it manually.
>* Instead use the provided comamnds to safely work with your state.
>>* `terraform state [subcommand] [options] [address]` 


## 1. Listing Resource in the State
The `list` subcommadn displays all resources addresses recorded in the terraform state file

```bash
$ terraform state list

aws_dynamodb_table.cars
aws_s3_bucket.finance-2020922
```


## 2. Showing Resource Details
To inspect details attributes fo a particular resource, use the `show` subcommand

```bash
$ terraform state show aws_s3_bucket.finance-2020922

resource "aws_s3_bucket" "terraform-state" {
  acl                         = "private"
  arn                         = "arn:aws:s3:::finance-2020922"
  bucket                      = "finance-2020922"
  bucket_domain_name          = "finance-2020922.s3.amazonaws.com"
  bucket_regional_domain_name = "finance-2020922.s3.us-west-1.amazonaws.com"
  force_destroy               = false
  hosted_zone_id              = "Z2F5ABCDE1ACD"
  id                          = "finance-2020922"
  region                      = "us-west-1"
  request_payer               = "BucketOwner"
  tags = {
    "Description" = "Bucket to store Finance and Payroll Information"
  }
}
versioning {
  enabled    = false
  mfa_delete = false
}
```

## 3. Moving Resources in the State

The `move` subcommand (or its alias `mv`) is used to change a resource's address within the state file.
-   The command is particularly useful when you want to rename a resource without recreating it.
    -   It requires specifying both a source address and a destination address

Consider the following example where you have a DynamoDB table defined in your Terraform configuration.
```bash
resource "aws_dynamodb_table" "state-locking" {
  name         = "state-locking"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }
}
```

And the corresponding state file entry appears as:
```bash
{
  "resources": [
    {
      "mode": "managed",
      "type": "aws_dynamodb_table",
      "name": "state-locking",
      "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]"
    }
  ]
}
```


To rename the resource from *state-locking* to *state-locking-db* **without having to recreate it**, execute:

```bash
$ terraform state mv aws_dynamodb_table.state-locking aws_dynamodb_table.state-locking-db

Move "aws_dynamodb_table.state-locking" to "aws_dynamodb_table.state-locking-db"
Successfully moved 1 object(s).
```


After moving the state, update your configuraiton file as shown below:
```bash
resource "aws_dynamodb_table" "state-locking-db" {
  name         = "state-locking-db"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }
}
```


When you run `terraform apply` afterward, Terraform should detect no changes for the renamed resource:

```bash
$ terraform apply
aws_dynamodb_table.state-locking-db: Refreshing state... [id=state-locking]
Apply complete! Resources: 0 added, 0 changed, 0 destroyed.

```

## 4. Pulling Remote State
When using remote state, the state fiel is not saved locally in your configuration directory.
-   Use the `pull` subcommand to download and print the remote state

```bash
$ terraform state pull
{
  "version": 4,
  "terraform_version": "0.13.0",
  "serial": 0,
  "lineage": "b6e2cf0e-ef8d-3c59-1e11-c6520dcd745c",
  "resources": [
    {
      "mode": "managed",
      "type": "aws_dynamodb_table",
      "name": "state-locking-db",
      "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
      "instances": [
        {
          "schema_version": 1,
          "attributes": {
            ...
          }
        }
      ]
    }
  ]
}
```

You can pipe the output to JSON query tools like `jq` for further data manipulation. For instance, to filter and display the `hash_key` attribute of the `state-locking-db` table, run:

```bash
$ terraform state pull | jq '.resources[] | select(.name == "state-locking-db") | .instances[].attributes.hash_key'
```


## 5. Removing Resources from the State

The `rm` (or remove) subcommand delete one or more resources from the Terraform state file. 
-   Removing a resource detaches it from Terraform management without destroying the actual infrastructure.

For example to remote S3 bucket resource from the state, run:
```bash
$ terraform state rm aws_s3_bucket.finance-2020922
Acquiring state lock. This may take a few moments...
Removed aws_s3_bucket.finance-2020922
Successfully removed 1 resource instance(s).
Releasing state lock. This may take a few moments...
```

>After removing a resource from the state, remember to also delete or comment out the corresponding resource block in your Terraform configuration file.
